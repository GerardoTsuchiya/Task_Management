Arquitectura General

El backend sigue la arquitectura MVC en capas que promueve NestJS: cada funcionalidad vive en su propio módulo con tres responsabilidades separadas.

C

Controller (Enrutador)

Recibe la petición HTTP. Extrae parámetros, cuerpo y usuario. No contiene lógica de negocio. Delega inmediatamente al Service.

Ejemplo: TasksController recibe POST /tasks y llama a tasksService.create(userId, dto).

S

Service (Lógica de Negocio)

Toda la lógica vive aquí: validaciones de acceso, reglas de negocio, construcción de queries. Llama a Prisma para interactuar con la BD.

Ejemplo: TasksService.create() verifica si el usuario pertenece al proyecto antes de crear la tarea.

P

PrismaService (Base de Datos)

Extiende PrismaClient. Es el único punto de contacto con PostgreSQL. Gestiona la conexión (onModuleInit/$connect) y la cierra (onModuleDestroy/$disconnect).

💡

NestJS usa Inyección de Dependencias: el Service se inyecta en el Controller a través del constructor. Esto lo hace el framework automáticamente al arrancar — tú solo declaras qué necesitas.

Ciclo de vida de una Petición

Antes de que cualquier código de negocio se ejecute, NestJS pasa la petición por varias capas:

1

Frontend → HTTP Request

El navegador envía POST http://localhost:3000/tasks con el header Authorization: Bearer <token> y un JSON body.

2

CORS Middleware

En main.ts, se configuró app.enableCors() con una expresión regular /^http:\/\/localhost(:\d+)?$/. Si el origen no es localhost, la petición es rechazada antes de llegar al código.

3

JwtAuthGuard Guard

Todos los controllers (excepto POST /auth/login y POST /auth/register) tienen el decorador @UseGuards(JwtAuthGuard). El Guard llama a JwtStrategy.validate():

  • Extrae el token del header Authorization: Bearer ...
  • Verifica la firma con JWT_SECRET
  • Busca el usuario en la BD con el sub (id) del payload
  • Si el usuario no existe → lanza 401 Unauthorized
  • Si es válido → adjunta el usuario al objeto request
4

ValidationPipe Pipe

Configurado globalmente en main.ts con tres opciones críticas:

  • whitelist: true → elimina cualquier campo del body que no esté declarado en el DTO
  • transform: true → convierte strings a números, instancia clases DTO
  • forbidNonWhitelisted: true → si viene un campo extra, retorna 400 Bad Request
5

Controller Method

El método correspondiente se ejecuta. Usa @GetUser() (decorador custom) para extraer el usuario del request, @Param(), @Body(), @Query(). Llama al Service.

6

Service Method

Ejecuta validaciones de acceso, construye la query de Prisma y la ejecuta. Puede lanzar NotFoundException, ForbiddenException, etc.

7

PrismaService → PostgreSQL

Prisma genera el SQL, lo ejecuta contra Supabase y retorna los datos tipados. El resultado sube por la cadena hasta el Controller, que lo serializa a JSON y lo envía como respuesta HTTP.

Autenticación JWT en Detalle

¿Qué es JWT?

JSON Web Token es un string codificado en base64 con tres partes: header.payload.signature. El servidor firma el token con un secreto (JWT_SECRET). Cualquier modificación al token invalida la firma.

El payload contiene: { sub: userId, email, iat (issued at), exp (expiration) }.

✅

Stateless: el servidor no guarda sesiones. Cada petición lleva su propio token. El servidor solo verifica la firma — no necesita consultar una tabla de sesiones.

JwtStrategy — Validación en cada request

// jwt.strategy.ts — se ejecuta en CADA petición protegida
async validate(payload: JwtPayload): Promise<AuthenticatedUser> {
  // payload.sub = userId guardado al hacer login
  const user = await this.prisma.user.findUnique({
    where: { id: payload.sub },
    select: { id, email, name, createdAt, updatedAt }
  });
  if (!user) throw new UnauthorizedException();
  return user; // se adjunta al request como req.user
}

@GetUser() — Decorador personalizado

// Extrae req.user directamente en el parámetro del controller
export const GetUser = createParamDecorator(
  (data, ctx) => ctx.switchToHttp().getRequest().user
);

// Uso en controller:
@Post()
create(@GetUser() user: AuthenticatedUser, @Body() dto: CreateTaskDto) {
  return this.tasksService.create(user.id, dto);
}

DTOs y Validación

Un DTO (Data Transfer Object) es una clase que define la forma del body esperado. Los decoradores de class-validator validan cada campo automáticamente.

// create-task.dto.ts
export class CreateTaskDto {
  @IsString()
  @MinLength(2)
  title!: string;               // obligatorio, mínimo 2 chars

  @IsOptional() @IsString()
  description?: string;          // opcional

  @IsOptional() @IsISO8601()
  dueDate?: string;              // formato "2026-05-25"

  @IsOptional() @IsEnum(TaskPriority)
  priority?: TaskPriority;        // solo LOW/MEDIUM/HIGH/URGENT

  @IsOptional()
  @Transform(({ value }) => value === null ? null : value)
  @Type(() => Number) @IsInt() @Min(1)
  projectId?: number;           // convierte "5" → 5 automáticamente

  @IsOptional() @Type(() => Number) @IsInt() @Min(1)
  assignedToId?: number;         // id del usuario al que se asigna
}

Si el body no cumple las reglas, el ValidationPipe retorna automáticamente 400 Bad Request con un mensaje descriptivo antes de llamar al Controller.

Flujo: Registro de Usuario

1

Frontend

POST /auth/register con body: { email, name, password }

2

ValidationPipe valida RegisterDto

Verifica que email sea válido, name no esté vacío y password tenga al menos 6 caracteres. Sin token requerido (ruta pública).

3

AuthService.register()

1. Busca si ya existe un usuario con ese email → si existe lanza 409 ConflictException.

2. Hashea la contraseña con bcrypt.hash(password, 10) (el 10 es el costo/sal).

3. Crea el usuario en la BD con prisma.user.create().

4. Genera un JWT con jwtService.signAsync({ sub: user.id, email }).

5. Retorna { accessToken, user } — sin la contraseña.

4

Frontend guarda el token

El frontend guarda accessToken en localStorage y lo incluye en todas las peticiones futuras.

Flujo: Login

1

Frontend

POST /auth/login con { email, password }

2

AuthService.login()

1. Busca el usuario por email. Si no existe → 401 Unauthorized.

2. Compara la contraseña con bcrypt.compare(plainPassword, hashedPassword). Si no coincide → 401.

3. Genera y retorna el JWT igual que en el registro.

🔒

El mensaje de error es siempre "Invalid credentials" tanto si el email no existe como si la contraseña es incorrecta. Esto es intencional para no revelar qué campo es incorrecto.

Flujo: Crear una Tarea

Este es el flujo más completo porque involucra validaciones de acceso, lógica de auto-asignación y la inserción en BD.

1

Frontend — api.post('/tasks', body)

Envía POST /tasks con header Authorization: Bearer TOKEN y body:

{ title, description, priority, dueDate, status, projectId, assignedToId }
2

JwtAuthGuard → JwtStrategy.validate()

Verifica el token, obtiene el usuario de la BD, lo adjunta al request.

3

ValidationPipe → CreateTaskDto

Valida campos, transforma projectId y assignedToId de string a número si vienen como query params.

4

TasksController.create()

@Post()
create(@GetUser() user: AuthenticatedUser, @Body() dto: CreateTaskDto) {
  return this.tasksService.create(user.id, dto);
}
5

TasksService.create() — Lógica de negocio

async create(userId, dto) {
  let assignedToId = dto.assignedToId ?? null;

  if (dto.projectId) {
    // Paso 5a: ¿tiene acceso al proyecto? (dueño o miembro aceptado)
    await this.ensureProjectAccessibleByUser(userId, dto.projectId);

    if (dto.assignedToId) {
      // Paso 5b: solo el dueño puede asignar a otros
      await this.ensureUserIsProjectOwner(userId, dto.projectId);
      // Paso 5c: el usuario asignado debe ser miembro
      await this.ensureUserIsMemberOfProject(dto.assignedToId, dto.projectId);
    } else {
      // Paso 5d: si es miembro (no dueño), auto-asignarse
      const isOwner = await prisma.project.findFirst({ where: { id, userId } });
      if (!isOwner) assignedToId = userId; // auto-asignación
    }
  }

  // Paso 5e: normalizar status/completed
  const completionData = this.buildCompletionData({ status: dto.status });

  // Paso 5f: insertar en BD
  return prisma.task.create({ data: { ...campos, userId, assignedToId } });
}
6

PostgreSQL — INSERT INTO "Task"

Prisma ejecuta el INSERT y retorna la tarea creada incluyendo sus relaciones (getTaskInclude()): proyecto, usuario asignado, subtareas y etiquetas.

7

Response → Frontend

El controller retorna el objeto tarea completo. HTTP 201 Created.

Función ensureProjectAccessibleByUser

Se llama en varias partes del servicio. Busca el proyecto asegurando que el usuario sea dueño O miembro aceptado:

private async ensureProjectAccessibleByUser(userId, projectId) {
  const project = await prisma.project.findFirst({
    where: {
      id: projectId,
      OR: [
        { userId },                                            // es el dueño
        { members: { some: { userId, status: 'accepted' } } }, // es miembro
      ]
    }
  });
  if (!project) throw new NotFoundException('Project not found');
}

Flujo: Completar / Descompletar una Tarea

El frontend llama a PATCH /tasks/:id/complete o PATCH /tasks/:id/uncomplete.

// TasksService.complete()
async complete(userId, id) {
  await this.ensureTaskBelongsToUser(userId, id); // verifica acceso
  return prisma.task.update({
    where: { id },
    data: {
      completed: true,
      status: TaskStatus.COMPLETED,
      completedAt: new Date()   // registra cuándo se completó
    }
  });
}

ensureTaskBelongsToUser

Verifica que el usuario sea el creador O el asignado de la tarea:

where: { id, OR: [{ userId }, { assignedToId: userId }] }

Flujo: Invitar a un Usuario al Proyecto

1

Frontend

POST /projects/:id/members con body { email: "usuario@ejemplo.com" }

2

ProjectsService.inviteMember(ownerId, projectId, dto)

Paso 1: Busca el proyecto con WHERE id = projectId AND userId = ownerId. Si no existe (o no es el dueño) → 404 NotFoundException.

Paso 2: Busca el usuario por email. Si no existe → 404.

Paso 3: Verifica que no se esté invitando a sí mismo → 400.

Paso 4: Crea un ProjectMember con status: 'pending'. Si ya existe (unique constraint) → 409 ConflictException.

Paso 5: Crea una Notification para el usuario invitado:

{ userId: invitedUser.id, type: 'project_invitation',
  title: 'Nueva invitación a proyecto',
  body: `${ownerName} te invitó a "${projectName}"`,
  data: { projectId, projectName, memberId } }
3

El invitado ve la notificación

En el Navbar, GET /notifications/invitations retorna las membresías con status: 'pending'. Aparece un badge rojo en el ícono de usuario.

Flujo: Aceptar una Invitación

1

Frontend

PATCH /notifications/invitations/:memberId/accept

2

NotificationsService.acceptInvitation(userId, memberId)

1. Busca ProjectMember WHERE id = memberId AND userId = userId AND status = 'pending'. Si no → 404.

2. Actualiza status = 'accepted'.

3. Marca la notificación relacionada como leída (read: true), buscándola por data.memberId = memberId.

3

Frontend recarga proyectos

Al aceptar, el frontend llama a loadData() que consulta GET /projects. El nuevo proyecto aparece en el sidebar porque la query incluye proyectos donde members.some({ userId, status: 'accepted' }).

Rechazar invitación

DELETE /notifications/invitations/:memberId/decline → NotificationsService.declineInvitation() elimina el registro ProjectMember directamente (no cambia status, lo borra).

Flujo: Visibilidad de Tareas (Filtro Complejo)

Esta es la parte más compleja del sistema. buildTaskFilters() construye una condición OR con tres casos para determinar qué tareas ve un usuario:

private buildTaskFilters(userId, query) {
  const where = {
    OR: [
      // Caso 1: el usuario creó la tarea
      { userId },

      // Caso 2: la tarea está asignada al usuario
      { assignedToId: userId },

      // Caso 3: la tarea pertenece a un proyecto
      // donde el usuario es dueño O miembro aceptado
      {
        projectId: { not: null },
        project: {
          OR: [
            { userId },                                              // es dueño
            { members: { some: { userId, status: 'accepted' } } }, // es miembro
          ]
        }
      }
    ]
  };
  // ...filtros adicionales de búsqueda, proyecto, prioridad, etc.
  return where;
}
EscenarioCondición que aplica
Tarea personal propiaCaso 1: userId
Tarea de proyecto, asignada a míCaso 2: assignedToId
Tarea de mi proyecto (soy dueño)Caso 3: project.userId
Tarea de proyecto ajeno donde soy miembroCaso 3: members.some(...)
⚠️

Prisma traduce este OR a un SQL con JOINs en una sola consulta. No son múltiples queries — es una sola consulta optimizada con índices.

Manejo de Errores

NestJS mapea automáticamente las excepciones a códigos HTTP:

ExcepciónHTTPCuándo se usa
NotFoundException404Recurso no encontrado o sin acceso (no se revela si existe)
UnauthorizedException401Token inválido, expirado o usuario no existe
ForbiddenException403Autenticado pero sin permiso (ej: asignar tarea sin ser dueño)
ConflictException409Email ya registrado, invitación duplicada, nombre de proyecto duplicado
BadRequestException400Validación fallida o lógica inválida (invitarse a sí mismo)

Errores de Prisma

Cuando ocurre una violación de restricción única en la BD, Prisma lanza un PrismaClientKnownRequestError con código P2002. El servicio lo captura y lo convierte en ConflictException:

if (error instanceof Prisma.PrismaClientKnownRequestError
    && error.code === 'P2002') {
  throw new ConflictException('...');
}
