Frontend Documentation

Aplicación web de gestión de tareas — React + TypeScript + Vite

El frontend es una Single Page Application (SPA) construida con React 18 y TypeScript. No utiliza un router externo; en su lugar maneja la navegación con estados de string en App.tsx: "login", "register", "dashboard" y "loading".

Toda la lógica de negocio (llamadas a la API, actualizaciones de estado) vive en DashboardPage.tsx, que actúa como contenedor principal y pasa datos hacia abajo mediante props. Los componentes hijos son puramente presentacionales.

Flujo de Autenticación

App.tsx es el responsable de manejar si el usuario está autenticado y qué página mostrar. Al cargar la aplicación ejecuta checkSession(). 

// App.tsx — estado de navegación
type Page = "loading" | "login" | "register" | "dashboard";

const [page, setPage] = useState<Page>("loading");
const [user, setUser] = useState<User | null>(null);

checkSession() — al montar la app

1
Busca el token en localStorage
localStorage.getItem("token"). Si no existe → muestra LoginPage.
2
Verifica el token con el backend
Llama a GET /auth/me. Si el backend lo acepta, devuelve los datos del usuario.
3
Resultado
Token válido → setUser(data), navega a dashboard.
Token inválido / expirado → localStorage.removeItem("token"), navega a login.

handleAuthSuccess(token, user)

Se llama tanto al hacer login como al registrarse exitosamente.

function handleAuthSuccess(token: string, user: User) {
  localStorage.setItem("token", token);   // guarda JWT
  setUser(user);                            // actualiza estado
  setPage("dashboard");                    // navega al dashboard
}

handleLogout()

function handleLogout() {
  localStorage.removeItem("token");   // elimina el token
  setUser(null);
  setPage("login");
}
El JWT no se decodifica en el frontend. Solo se almacena y se envía en el header Authorization: Bearer <token>. El backend es quien lo valida en cada petición.

Servicio API

Toda comunicación con el backend pasa por services/api.ts. Es una pequeña capa de abstracción sobre fetch que añade automáticamente el token de autenticación a cada request. 

// services/api.ts
const BASE_URL = "http://localhost:3000";

function getHeaders() {
  const token = localStorage.getItem("token");
  return {
    "Content-Type": "application/json",
    ...(token ? { "Authorization": `Bearer ${token}` } : {}),
  };
}

Métodos disponibles

MétodoFirmaHTTP

api.post(endpoint, body)(url: string, body: object) → Promise<any>POST
api.patch(endpoint, body?)(url: string, body?: object) → Promise<any>PATCH
api.delete(endpoint)(url: string) → Promise<any>DELETE

Manejo de errores

Si el servidor devuelve un status que no sea 2xx, el servicio lanza una instancia de ApiError que incluye el status HTTP. Esto permite que los componentes hagan manejo específico por código de error: 

class ApiError extends Error {
  status: number;
  constructor(message: string, status: number) {
    super(message);
    this.status = status;
  }
}

// Ejemplo de uso en un componente
try {
  await api.post(`/projects/${id}/members`, { email });
} catch (err: any) {
  if (err.status === 404) setError("Usuario no encontrado.");
  else if (err.status === 409) setError("Ya es miembro o tiene invitación.");
}

Respuesta vacía

Si el backend responde con 204 No Content (ej. al eliminar una tarea), el servicio devuelve null en lugar de intentar parsear JSON vacío.

Funciones clave

// Carga invitaciones pendientes desde el backend
async function loadInvitations() {
  const data = await api.get("/notifications/invitations");
  setInvitations(data);
}

// Se ejecuta al montar y cuando cambia el usuario
useEffect(() => { loadInvitations(); }, [user]);

// Acepta una invitación y recarga el dashboard
async function handleAccept(id: number) {
  setLoadingId(id);
  await api.patch(`/notifications/invitations/${id}/accept`);
  await loadInvitations();
  onInvitationAccepted?.();   // recarga proyectos y tareas en el dashboard
  setLoadingId(null);
}

// Rechaza una invitación
async function handleDecline(id: number) {
  setLoadingId(id);
  await api.delete(`/notifications/invitations/${id}/decline`);
  await loadInvitations();
  setLoadingId(null);
}

Badge de invitaciones

Si hay invitaciones pendientes (invitations.length > 0), aparece un círculo rojo con el número en la esquina superior derecha del icono de usuario. Solo se muestra cuando hay invitaciones.

Detección de clic fuera (click-outside)

Usa useRef + useEffect para detectar clics fuera del dropdown y cerrarlo automáticamente: 

useEffect(() => {
  function handleClickOutside(e: MouseEvent) {
    if (dropdownRef.current && !dropdownRef.current.contains(e.target as Node)) {
      setIsOpen(false);
    }
  }
  document.addEventListener("mousedown", handleClickOutside);
  return () => document.removeEventListener("mousedown", handleClickOutside);
}, []);

loadData() — carga inicial y recarga

async function loadData() {
  const projs = await api.get("/projects");
  setProjects(projs);
  const allTasks = await api.get("/tasks");
  setTasks(allTasks);
}

Se ejecuta al montar el componente y después de cualquier operación exitosa.

Lógica de filtrado de tareas

// 1. Filtrar por proyecto activo
const projectTasks = activeProject
  ? tasks.filter(t => t.projectId === activeProject.id)
  : tasks;

// 2. Opcionalmente filtrar por "asignadas a mí"
const filteredTasks = filterAssignedToMe && user
  ? projectTasks.filter(t => t.assignedTo?.id === user.id)
  : projectTasks;

// 3. Separar en tres secciones
const todayStr = new Date().toISOString().split("T")[0];

filteredTasks.forEach(task => {
  const taskDate = task.dueDate?.split("T")[0];
  if (task.status === "COMPLETED") completedTasks.push(task);
  else if (taskDate && taskDate < todayStr) overdueTasks.push(task);
  else pendingTasks.push(task);
});

Patrón de actualización optimista

Para una mejor experiencia de usuario, el dashboard actualiza el estado local antes de que el servidor confirme la operación. Si la petición falla, revierte al estado anterior. 

// Ejemplo: eliminar tarea con actualización optimista
async function handleDeleteTask(id: number) {
  const previousTasks = [...tasks];          // guarda estado previo
  setTasks(prev => prev.filter(t => t.id !== id)); // actualiza UI inmediatamente

  try {
    await api.delete(`/tasks/${id}`);
    await loadData();                          // recarga para sincronizar
  } catch {
    setTasks(previousTasks);                  // revierte si hay error
  }
}

Flujo Completo: Crear Tarea

1
Usuario hace clic en "+" en el header
setShowModal(true) → se renderiza <NewTaskModal />
2
El modal recibe props de contexto
projectId: ID del proyecto activo o "0"
isOwner: activeProject.userId === user.id
projectMembers: miembros del proyecto excluyendo al usuario actual
3
Usuario llena el formulario y hace clic en "Crear Task"
Validación frontend: título ≥ 2 chars, fecha requerida, prioridad requerida
4
onSave() llama a handleSaveTask() en DashboardPage
Se crea una tarea optimista con ID negativo (Date.now() * -1) y se añade al estado
5
POST /tasks enviado al backend
{
  title, description, priority, dueDate,
  status: "PENDING",
  projectId: activeProject?.id ?? null,
  assignedToId: assignedToId ?? null
}
6
Backend procesa la lógica de asignación
Si el usuario NO es dueño → auto-asigna la tarea al creador.
Si es dueño y eligió asignado → asigna a ese usuario.
7
Éxito: loadData() recarga todo
La tarea optimista es reemplazada por la real con ID del servidor. Error: se elimina la tarea optimista.

Flujo Completo: Completar/Descompletar Tarea

1
Usuario hace clic en el círculo de estado en TaskRow
Se llama a onToggleStatus(task) → toggleTaskStatus(task) en DashboardPage
2
Actualización optimista
Se invierte el status en el estado local inmediatamente para feedback visual instantáneo
3
API call según el estado actual
Si era COMPLETED → PATCH /tasks/{id}/uncomplete
Si era PENDING o IN_PROGRESS → PATCH /tasks/{id}/complete
4
Recarga y reorden de tareas
loadData() recarga, la tarea aparece en la sección correcta (Completadas / Pendientes)

Patrones Clave del Frontend

Actualización Optimista

El estado local se actualiza inmediatamente antes de que la API confirme la operación. Si la API falla, se revierte el estado. Esto hace la UI sentirse instantánea.

Routing por Estado

No hay react-router. La navegación entre pantallas es simplemente setPage("dashboard"). App.tsx renderiza condicionalmente según el valor de page.

Token en localStorage

El JWT se guarda en localStorage al hacer login. Se lee en cada petición de la API. Al hacer logout se elimina.

Props Drilling

El estado vive en DashboardPage y se pasa hacia abajo mediante props. No hay Context ni estado global. Funciona bien para esta escala de app.

Modales con Backdrop

Todos los modales usan position: fixed; inset: 0 con fondo semi-transparente. Haciendo clic en el backdrop (fuera del modal) se cierra.

Estilos Inline

No hay archivos CSS separados para los componentes del dashboard. Todos los estilos son objetos inline con valores de COLORS. Evita colisiones de CSS.
